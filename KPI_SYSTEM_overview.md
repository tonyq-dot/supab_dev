Техническая спецификация: Система расчета KPI и бонусов (VFX/Animation)
1. Общее описание
Система предназначена для учета выполненных работ 3D-аниматоров, автоматического расчета баллов (KPI) и формирования итоговой квартальной премии на основе оклада и коэффициентов эффективности.

Стек: Supabase (PostgreSQL, Auth, Edge Functions) + React (Frontend).

2. Структура Базы Данных (Схема)
Для реализации в Supabase потребуется создать следующие сущности (таблицы):

2.1. Справочники (Dictionaries)
Эти таблицы редко меняются, но должны быть редактируемыми для администраторов.

Table: project_types (Типы проектов)
Используется для определения коммерческой значимости проекта.

Тип (Name)	Коэффициент (Value)	Описание
ШОУ	2.0	Коммерческий проект, прибыль
РАЗРАБОТКА	1.5	Превью для клиента/концепт
ПРЕВЬЮ	1.5	Создание инструментов и оптимизация
НАСТАВНИЧЕСТВО	1.0	Обучение коллег
ИННОВАЦИИ	1.5	Технические инновации
ПРОМО	1.0	Некоммерческий проект
Table: drone_ranges (Диапазоны дронов)
Коэффициент масштаба. Если число дронов попадает в диапазон [min...max], берется коэффициент.

Min	Max	Коэффициент
200	500	1.00
500	1000	1.15
1000	3000	1.30
3000	5000	1.50
5000	8000	1.75
10000	15000	2.00
Table: work_types (Виды работ)
Базовая стоимость единицы работы.
Примеры: 2Д Сцена (4.5), 3Д Сцена (7.5), Полет группами (14.5) и т.д. Полный список см. в Excel.

Table: rework_coefs (Переработка)

Статус	Значение
ДА	1.5
НЕТ	1.0
2.2. Операционные таблицы
Table: projects (Проекты)
Заполняется Менеджером. Аниматор только выбирает проект из списка.

id (UUID, PK)

name (String) — Название (напр. "Дубай Аэрошоу")

type_id (FK -> project_types)

drone_count (Integer) — Количество дронов

is_rework (Boolean) — Переработка (FK -> rework_coefs)

created_at (Timestamp)

status (Enum: Active/Archived)

Table: work_logs (Журнал работ)
Заполняется Сотрудником. Основная таблица для расчета.

id (UUID, PK)

user_id (UUID, FK -> auth.users) — Кто сделал

project_id (UUID, FK -> projects) — Ссылка на проект

work_type_id (UUID, FK -> work_types) — Что сделал

quantity (Integer) — Количество сцен/элементов

date (Date) — Дата выполнения

image_url (String, Optional) — Ссылка на превью (из Storage)

Table: period_configs (Настройки периода)
Глобальные константы, которые могут меняться каждый квартал.

period_name (String) — напр. "Q3 2026"

norm_points (Integer) — Норма баллов (сейчас: 500)

company_profit_coef_q1 (Float) — Коэффициент прибыли компании (Q1) (сейчас: 0.3)

Total Points = Сумма всех Score сотрудника за выбранный период (квартал).

4. Пользовательские сценарии (UX)
Роль: Аниматор (Сотрудник)
Логин в систему.

Видит "Дашборд":

Текущие баллы за месяц/квартал.

Прогноз премии (в рублях).

Нажимает "Добавить работу":

Выбирает Проект (из списка активных).

Выбирает Вид работы (из списка).

Вводит Количество.

(Опционально) Загружает скриншот/превью.

Важно: Аниматор НЕ вводит коэффициенты дронов или типа проекта вручную. Они подтягиваются автоматически от Проекта.

Роль: Менеджер
Управление проектами: Создает новый проект, задает ему параметры (Шоу/Промо, кол-во дронов).

Настройка квартала: Устанавливает Q1 (коэфф. прибыли компании) на текущий квартал.

Отчетность: Видит сводную таблицу по всем сотрудникам (кто сколько набрал баллов).

5. Рекомендации по реализации в Supabase
Row Level Security (RLS):

Настроить RLS так, чтобы сотрудник мог видеть (SELECT) и добавлять (INSERT) только свои записи в work_logs.

Менеджер должен иметь доступ ко всем записям.

Database Views:

Создайте SQL View detailed_scores, который делает JOIN всех таблиц (Logs, Projects, Coefs) и сразу выдает посчитанный балл для каждой строки. Это упростит запросы с фронтенда.

Images:

Используйте Supabase Storage (bucket work-previews). В таблицу work_logs сохраняйте только путь к файлу.
